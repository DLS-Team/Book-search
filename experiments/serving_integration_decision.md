# Serving Integration Decision (Task 4.4)

## 1. Архитектурный паттерн и философия модуля
Для реализации единого search interface (`src/search_engine.py`) применен паттерн **Facade (Фасад)**. Модуль не содержит логики базового ранжирования (BM25, Dense, RRF). Он выступает маршрутизатором: принимает запрос, выбирает нужный модуль (Role 1, 2 или 3), получает сырые данные и приводит их к единому выходному контракту (схеме раздела 2.3 плана).

**Отказ от Graceful Degradation (Заглушек):** В финальной версии применен принцип **Fail-Fast**. Все `try/except` с Mock-объектами удалены. Если хоть один модуль другой роли не готов (не экспортирует нужную функцию), импорт `search_engine.py` упадет с ошибкой, и веб-сервер не стартует. Это исключает ситуацию, когда демо работает, но показывает фейковые данные, скрывая проблемы интеграции в команде.

**Модуль как библиотека:** `search_engine.py` не является точкой входа в приложение (self-contained скриптом). Это библиотека, которая предоставляет два публичных API для Role 2 (Бэкенд): функцию `initialize_server_state()` и функцию `search()`. Блок `if __name__ == "__main__"` существует исключительно для локального дебага и эмуляции работы сервера без поднятия веб-интерфейса.

## 2. Зона ответственности Role 4 в поиске (Метод `dense_ann`)
Так как построение ANN индекса (HNSW) является задачей Role 4 (Таска 4.1), функция поиска по этому индексу реализована внутри `search_engine.py`. 
Алгоритм метода `"dense_ann"`:
1. Вызов функции `encode_query()` из модуля Role 2 для получения вектора запроса.
2. L2-нормализация вектора (для корректного Inner Product).
3. Вызов `APP_STATE["faiss_hnsw"].search()` (поиск по графу HNSW в RAM).
4. **Критический шаг (Маппинг ID):** FAISS возвращает числовые индексы (0, 1, 2...). Чтобы система могла найти текст чанка, эти индексы конвертируются в строковые ID (например, `"41496-8_chunk_000001"`) с использованием файла `chapter_ids.json`, который предварительно загружается в оперативную память.

## 3. Интеграция контрактов других ролей (Обработка данных)

Поскольку разные роли возвращают результаты в разных форматах, `search_engine.py` выполняет роль "адаптера" перед финальным форматированием:

*   **Role 1 (BM25):** Ожидается, что возвращает список стандартных словарей `dict`. Передается в `_format_result()` напрямую.
*   **Role 2 (Dense):** Возвращает объекты `DenseResult` (dataclass). Модуль использует `dataclasses.asdict()` для конвертации объекта в словарь "на лету", чтобы не ломать логику `_format_result()`.
*   **Role 3 (Hybrid RRF):** Внутри своей функции Role 3 уже собирает финальные словари и умно подтягивает метаданные книг от Role 1. Результаты принимаются напрямую.
*   **Важное замечание по путям:** Дефолтные пути к индексам у других ролей могут не совпадать с актуальной структурой репозитория (например, Role 3 ищет BM25 в `outputs/`). Поэтому при вызове `search_hybrid_rrf` мы **явно переопределяем** аргумент `bm25_index_dir`, передавая реальный путь из констант `search_engine.py`.

## 4. Жизненный цикл приложения (Offline vs Online)

Модуль строго разделяет тяжелые операции ввода-вывода (I/O) и быстрые вычисления, чтобы соответствовать требованиям к latency онлайн-сервиса:

1.  **Фаза Bootstrap (Offline):** Вызов `initialize_server_state()` при старте веб-сервера.
    *   Загрузка `processed_chapters.jsonl` в словарь Python (через `load_chunks_db` из Таски 4.2).
    *   Загрузка маппинга `chapter_ids.json` в список.
    *   Загрузка HNSW графа в память через `safe_read_index()` (фикс бага с кириллицей в пути).
2.  **Фаза Online Request:** Вызов `search()` при HTTP-запросе пользователя.
    *   Никаких обращений к диску. Все операции (поиск по FAISS, словарным ключам, слияние) происходят строго в RAM со скоростью миллисекунды.

## 5. Интеграция Quality Gate (Таска 4.5)
Сразу после форматирования результатов в единую схему, список прогоняется через `apply_quality_gate()`. 
Проверяется `score_or_rank` первого результата на предмет падения ниже эвристических порогов (разных для BM25, Dense и Hybrid). При провале все результаты помечаются флагом `low_confidence: True` и текстовым предупреждением для фронтенда Role 2. Provenance при этом сохраняется (согласно этическим ограничениям плана).

---

## 6. Реализация каскада (Retrieve -> Resolve)
В методе `search(method="refined")` реализована логика двухэтапного пайплайна:
1. Вызов базового поиска (обычно `hybrid` от Role 3), который возвращает топ релевантных ID чанков.
2. Передача списка ID в `resolve_fragments()` (Таска 4.2).
3. Функция разрешения обогащает словари полем `best_fragment` (доставая тексты из заранее загруженного в RAM словаря `_CHUNK_DB`) и метаданными книги.
4. Результат передается на фронтенд.

## 7. Итог
Файл `search_engine.py` является финальной сборочной точкой (Glue Code) проекта. Он берет разрозненные алгоритмы команд, нормализует их выходные данные, управляет состоянием оперативной памяти и предоставляет единую, понятную функцию `search()`, готовую к подключению к веб-серверу.
