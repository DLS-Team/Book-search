# Serving Integration Decision (Task 4.4)

## 1. Архитектурный паттерн и философия модуля
Для реализации единого search interface (`src/search_engine.py`) применен паттерн **Facade (Фасад)**. Модуль выступает маршрутизатором: принимает запрос, выбирает нужный модуль (Role 1, 2 или 3), получает сырые данные, нормализует их, обогащает текстом и приводит к единому выходному контракту.

**Fail-Fast с Graceful Degradation для ANN:** 
Базовые модули (BM25, Dense, чанки) работают по принципу Fail-Fast: если они не импортируются, сервер упадет при старте. Однако HNSW индекс (Task 4.1) вынесен в опциональную зону. Если файл `faiss_hnsw.index` или `chapter_ids.json` отсутствует/поврежден, сервер **стартует**, складывает предупреждение в `APP_STATE["warnings"]` и позволяет работать методам `bm25` и `dense`. При попытке вызова `dense_ann` или `hybrid` пользователь получит понятную ошибку `RuntimeError("HNSW index or chapter-ID mapping is not loaded...")`.

**Модуль как библиотека:** `search_engine.py` предоставляет два публичных API: `initialize_server_state()` и `search()`. Блок `if __name__ == "__main__"` существует для локального дебага.

## 2. Универсальный адаптер данных (`_to_dict` и `_format_result`)
Поскольку разные роли возвращают результаты в разных форматах (dataclass, Pydantic model, namedtuple, dict), реализована функция `_to_dict()`. Она использует утиную типизацию (`is_dataclass`, `model_dump`, `_asdict`), чтобы привести любой вход к стандартному словарю "на лету" без изменения исходного кода модулей Role 1/2/3.

Далее `_format_result()` приводит данные к единой схеме:
- Извлекает `chapter_id`, `paragraph_position`, `book`, `author`.
- Извлекает скор, проверяя по очереди поля `score`, `similarity`, `rrf_score`.
- Формирует строку `provenance` (Gutenberg ID, Chapter ID, Para Pos) для этического прослеживания источника.

## 3. Зона ответственности Role 4 в поиске (Метод `dense_ann`)
Алгоритм метода `"dense_ann"`:
1. Векторизация запроса через `encode_query()`.
2. L2-нормализация вектора (для корректного Inner Product).
3. Поиск по графу HNSW в RAM (`APP_STATE["faiss_hnsw"].search()`).
4. **Маппинг ID:** FAISS возвращает числовые индексы. Функция `_chapter_id_at()` конвертирует их в строковые ID, поддерживая как list, так и dict форматы из `chapter_ids.json`.

## 4. Жизненный цикл приложения (Offline vs Online) и Кэширование

1.  **Фаза Bootstrap (Offline):** Вызов `initialize_server_state()`.
    *   Загрузка `processed_chapters.jsonl` (`load_chunks_db`).
    *   Прелоадинг BM25 и ресурсов точного Dense поиска.
    *   Безопасное чтение HNSW графа через `safe_read_index()` (копирование во временный ASCII-файл для обхода бага FAISS с кириллицей в путях).
2.  **Фаза Online Request:** Вызов `search()`.
    *   **Кэширование:** Реализовано через `@lru_cache(maxsize=256)` на уровне `_cached_search()`. При повторных одинаковых запросах система мгновенно отдает результат из RAM.
    *   **Deep Copy:** Функция `search()` возвращает `copy.deepcopy()` закэшированных данных. Это критически важно, так как фронтенд/бэкенд может мутировать словари результатов, и мы не можем допустить порчи кэша.

## 5. Реализация каскада (Retrieve -> Resolve) для ВСЕХ методов
Ранее двухэтапный пайплайн применялся только для метода `refined`. В новой архитектуре **любой** метод поиска проходит через этап резолва текста:
1. Базовый поиск возвращает топ релевантных ID чанков (и скоры).
2. Список передается в `resolve_fragments()`.
3. Функция обогащает словари полем `best_fragment` (доставая тексты из RAM) и метаданными книги.
4. Происходит слияние (Merge): сохраняются оригинальные скоры и ранги, но переопределяются текстовые поля.
Это гарантирует, что фронтенд всегда получит конкретный текст главы, а не просто ID.

---

## 6. Оптимизация Hybrid поиска (Concurrency)
Метод `hybrid` использует `reciprocal_rank_fusion()`. Для минимизации latency оба ветви поиска (BM25 и Dense ANN) запускаются **параллельно** с помощью `ThreadPoolExecutor(max_workers=2)`. Поскольку поиск FAISS и BM25 в Python отпускает GIL (Global Interpreter Lock) во время вычислений на C++, это дает реальный прирост скорости.

### Основные изменения, которые я вынес в документацию:
1. **Кэширование:** Документировал использование `lru_cache` и обязательный `deepcopy` при возврате (защита от мутаций).
