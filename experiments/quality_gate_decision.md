# Quality Gate Decision (Task 4.5)

## 1. Суть компонента
Реализован механизм "абстенции" (abstention) в `search_engine.py`. Если система находит результаты, но они выглядят нерелевантными (низкий скор или ошибка выделения фрагмента), система не выдает их как уверенный ответ, а помечает специальным флагом `low_confidence: True` и текстовым предупреждением `warning`.

## 2. Инженерное решение: Динамические пороги и валидация текста
Вместо сложных вероятностных моделей использована функция `apply_quality_gate()`, которая проверяет топ-результат на соответствие жестким порогам (thresholds) в зависимости от метода поиска, а также валидирует успешность резолва текста.

Обновленные пороги (константа `QUALITY_GATE_THRESHOLDS`):
- **BM25 (< 1.5):** Низкий TF-IDF скор.
- **Dense / Dense ANN (< 0.20):** Косинусное сходство ниже 0.20 говорит о слабом смысловом совпадении. (Снижено с 0.4, так как фиксированный порог оказался слишком жестким для реального распределения оценок эмбеддингов и вызывал ложные срабатывания).
- **Hybrid / Refined (< 0.01):** Результаты Reciprocal Rank Fusion (RRF) математически представляют собой дроби вида `1 / (k + rank)`, поэтому их значения изначально очень малы. Порог 0.01 отсеивает только абсолютно нерелевантные склейки.

**Дополнительная проверка `refined`:** Для метода `refined` (и других) проводится проверка поля `best_fragment`. Если внутри текста фрагмента обнаруживается подстрока `[Error:`, система принудительно ставит `low_confidence = True` с предупреждением `"Fragment resolution failed."`, даже если скор выглядит высоким.

**Почему не LLM-as-judge:** Использование LLM для оценки каждого запроса добавило бы 1-3 секунды latency и потребовало API ключей. LLM-as-judge предназначен для offline-оценки качества датасетов, а не для online serving.

## 3. Сохранение Provenance (Ethical Controls)
В соответствии с разделом 4.13 плана, при срабатывании quality gate результаты **не удаляются**. Фронтенд получит полный список с книгами, авторами и provenance, но увидит флаг `low_confidence=True`. 
Это позволяет пользователю самому принять решение о том, кликнуть на сомнительный результат или нет, избегая "черного ящика".

## 4. Интеграция с фронтендом (Role 2)
Модуль `search_engine.py` возвращает список словарей, в которых обязательно присутствуют поля `low_confidence` (boolean) и `warning` (string или None). 
Задача Role 2 — прочитать этот флаг и, если он `True`, отрендерить плашку: *"Система не смогла найти точного совпадения по вашему запросу. Результаты ниже могут быть неточными."*

## 5. Тестирование на категории "Ambiguous / Weak evidence"
При прогоне на запросе из пулла Role 3 (например: *"a scene that feels like a dream inside a dream"*), если базовый поиск вернет скор ниже указанных порогов или произойдет сбой при доставании текста чанка, quality gate успешно перехватит этот ответ до того, как он дойдет до пользователя в качестве "уверенного".
