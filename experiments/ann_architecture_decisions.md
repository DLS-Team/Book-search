# ANN Index Architecture Decision: HNSW vs IVF

## Decision

В качестве serving-ориентированного ANN-индекса выбран FAISS HNSW (Hierarchical Navigable Small World).

## Alternatives Considered

1. FAISS IVF (Inverted File Index): Требует явного этапа обучения (clustering/кластеризация) при построении. Поиск зависит от параметра nprobe (сколько кластеров проверять).
2. FAISS IVF-PQ (с компрессией): Добавляет проблему потери качества из-за квантизации продукта (Product Quantization).

## Why HNSW?

1. Предсказуемый p95 latency: HNSW строит граф связей, что дает более стабильное время поиска на хвосте распределения (p95), в то время как IVF может резко замедляться на "сложных" запросах, попавших в плохие кластеры. Это критично для веб-демо (Таска 2.8), где нельзя допускать долгих зависаний.
2. Не требует обучения: Можно добавлять новые векторы без переобучения кластеров (хотя в нашем офлайн-пайплайне это не критично, это хорошая инженерная практика).
3. Лучшая точность при сопоставимом RAM: Для корпуса в ~500k объектов HNSW легко укладывается в оперативную память даже без сжатия, обеспечивая Recall > 0.95 "из коробки".

## Measured Trade-offs (будет заполнено после прогона на реальных данных)
- Ожидаемый риск: HNSW требует больше RAM (хранит граф) и дольше строится по сравнению с IVF.
- Метрики для оценки: Build time, Index size on disk, RAM usage, p50/p95 latency, Recall@10 против FAISS Flat.

## Tuning Parameters
- M (количество соседей в графе): контролирует баланс между плотностью графа и памятью. Дефолт: 32.
- efConstruction (качество построения): время сборки vs качество поиска. Дефолт: 200.
- efSearch (качество поиска на инференсе): Главный рычаг trade-off между latency и recall во время онлайна. Будет подбираться на сэмпле
