# ANN Indexing Decision & Benchmark Results (Task 4.1)

## 1. Выбранный алгоритм
Для serving-кандидата выбран **FAISS HNSW (Hierarchical Navigable Small World)** с метрикой Inner Product (векторы L2-нормализованы Role 2, `dim=384`).

## 2. Обоснование альтернатив
- **IVF (Inverted File Index)**: Отклонен. Требует этапа обучения (k-means), сложнее в настройке параметра `nprobe` для обеспечения стабильного p95 latency. HNSW дает более предсказуемое время поиска за счет графовой структуры, что критично для веб-демо (Role 2, Таска 2.8).
- **Компрессия (PQ / Float16)**: Отклонена. Индекс HNSW на реальных данных (см. таблицу) занимает ~1.05 GB RAM. Это полностью укладывается в стандартные лимиты демо-стенда (8-16 GB RAM), внедрение сжатия только понизит Recall без видимой выгоды по памяти.

## 3. Техническая реализация (Engineering Note)
При интеграции FAISS на Windows обнаружен баг: C++ ядро FAISS (`write_index` / `read_index`) не поддерживает пути, содержащие кириллицу (например, папки пользователя OneDrive `C:\Users\Кирилл\...`). 
**Решение:** Индексы создаются через `tempfile.mkstemp()` в ASCII-пути системной директории Temp, а затем переносятся в целевую папку проекта средствами Python (`shutil.move`). Для чтения используется безопасная обертка `safe_read_index`, копирующая индекс в Temp перед передачей в C++ ядро FAISS. Это гарантирует кроссплатформенность пайплайна.

## 5. Итоговый вывод (Trade-off)
Мы жертвуем **~240 МБ оперативной памяти** (1.05 GB против 0.81 GB у Flat) и **~4 минутами** офлайн-сборки индекса (которая происходит один раз), чтобы получить колоссальное снижение p95 latency **на 96%** (с 56.7 мс до 2.1 мс). 

Потеря точности (Recall=98.8%) составляет всего **1.2%**. Это абсолютно приемлемая плата за ускорение, так как на этапе гибридного поиска (Role 3) эта небольшая потеря dense-кандидатов будет с лихвой скомпенсирована результатами от лексического поиска BM25.

**Отказ от HNSW (Fast):** Вариант "Fast" был отклонен как основной для продакшена. Несмотря на невероятную скорость (p95 = 0.4 мс), он теряет более 10% релевантных документов (Recall = 89.8%). Для поисковой системы такой процент потерь на этапе выдачи кандидатов критичен, поэтому для `search_engine.py` (Таска 4.4) выбран индекс `faiss_hnsw.index` (Balanced).
