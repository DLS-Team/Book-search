# Paragraph/Chunk Resolution Decision (Task 4.2)

## 1. Архитектурное изменение парадигмы
Изначально Таска 4.2 предполагала "Small-to-Big retrieval": поиск целой главы базовым алгоритмом с последующим локальным TF-IDF поиском нужного абзаца внутри неё.

**Реальное решение:** По результатам подготовки данных (Role 1), searchable-объектом выбран **чанк (chunk)**. Чанк эквивалентен абзацу или небольшому фрагменту текста. Следовательно, базовый поиск (BM25 / FAISS) **уже** ищет на нужном нам уровне гранулярности. 

## 2. Суть компонента в новых реалиях
Модуль `paragraph_refinement.py` преобразован в **Metadata Resolver** (Резолвер метаданных).
Единственная задача функции `resolve_fragments()` — взять список `chapter_id`, возвращенных базовым поиском, и сопоставить их с реальными текстами и авторами из файла `processed_chapters.jsonl`.

## 3. Отказ от локального TF-IDF
Использование `scikit-learn` (TfidfVectorizer) внутри пайплайна было **отменено**.
**Обоснование:** Базовый поиск (Dense или BM25) уже отранжировал чанки по смысловому совпадению с запросом. Если мы возьмем топ-1 чанк от FAISS, его текст и есть наш `best_fragment`. Запускать второй алгоритм ранжирования по тексту, который уже выиграл у 500 тысяч других текстов, бессмысленно — это только исказит результаты и добавит задержку.

## 4. Реализация и Latency
- **Хранилище:** Файл `processed_chapters.jsonl` (от Role 1) загружается в память Python (в словарь `dict`) один раз при старте сервера функцией `load_chunks_db()`.
- **Latency онлайн-поиска:** Поиск ключа в словаре Python имеет сложность O(1). Добавленная latency Таски 4.2 составляет **< 0.1 мс** на 10 результатов. Это накладные расходы исключительно на извлечение строк из памяти.

## 5. Интеграция в Search Engine (Таска 4.4)
В `search_engine.py` метод `"refined"` теперь работает так:
1. Вызывает базовый поиск (например, Hybrid RRF), получая топ чанков с их `score` и `chapter_id`.
2. Передает этот список в `resolve_fragments()`.
3. Функция обогащает словари полем `best_fragment` (текст чанка) и метаданными книги.
4. Результат отправляется на фронтенд.

## 6. Ограничение (Limitation)
Так как мы отображаем чанк "как есть", если чанк оказался обрезан на середине предложения (из-за логики разбиения Role 1), пользователь увидит обрывок. Это приемлемый trade-off для поисковой выдачи (как в Google), но в идеальном мире требовалось бы "умное" слияние границ чанков (что выходит за рамки текущего спринта).
